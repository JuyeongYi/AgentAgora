## A. 미회신 원인 (Inst3/Inst6 자가 진단)

두 사람 모두 **큐 적체나 응답 거부가 아니라 "폴링 공백 / consumer down"**:

- **Inst6**: "Stop hook 자동 agora.wait가 실제로는 사용자 수동 트리거 의존 상태였음. 12:26 broadcast가 큐에 쌓였지만 wait poll이 열려있지 않아 수신 안 됨"
- **Inst3**: "내 직전 wait 종료가 12:26:02, broadcast가 12:26:09 — 7초 차이의 폴링 공백. 다음 wait에서 두 task 한 번에 수신"
- **Inst3 통찰**: **운영 모드 비대칭** — 자동 stop-hook 루프로 도는 인스턴스 vs 수동 트리거 기반 인스턴스의 응답성 격차. "Inst1이 catch-up 보낸 사실 자체가 이 비대칭의 신호"

→ **5번째 갭 추가: 수신자 활성 상태 가시성 부재**

## B. 코드 수정 — 워커 다수 합의(전원 일치)

| 변경 | 합의 내용 | 비고 |
|---|---|---|
| **`conversation_id` 신규 필드** | 첫 dispatch가 UUID 생성 / `in_reply_to` chain 자동 상속 / 동시 교차는 별개 ID + 명시 머지 옵션 | 7명 전원 동의 |
| **종료 플래그** (`closing` / `final` / `terminal` bool) | 양방향 도달 시 closed, 단방향은 half-closed. 자연어 의존 폐지 | 7명 전원, 명명만 차이 |
| **`agora.instances` 부하 메타** (`inbox_depth`, `in_flight`, `last_seen`) | dispatch 전 사전 점검 가능 | 7명 전원 |
| **`priority` + `agora.wait(sort=...)`** | FIFO 기본 + priority 옵션, 동률은 `command_id` 사전순 tie-breaker | 6명 동의, Inst4만 "docstring 수정만으로 충분" |

## C. 의견 분기점

| 축 | 미니멀리스트 (Inst4) | 풀스택 (Inst8) |
|---|---|---|
| 신규 도구 | **0개** — 기존 시그니처에 필드 4개만 추가 | `agora.peek`, `agora.conversation_status`, `agora.conversations_list` 3개 |
| 서버 state | 없음(envelope 수준) | conversation 테이블 신설 (in-memory → sqlite) |
| 응답 우선순위 | FIFO를 docstring으로 명문화하면 끝 | priority enum + sort 파라미터 |

**Inst7(tester)의 결정적 보강**: priority 동률 + 같은 created_at일 때 `command_id` 사전순까지 tie-breaker로 박아야 결정적(flaky 차단). peek은 "advisory only, atomicity 보장 안 함" 명시 필요 (TOCTOU race).

## G. 즉시 실행 가능한 권장 (수렴 합의 기준)

1. **`agora.dispatch`에 옵션 4종 추가**: `conversation_id`, `closing`, `priority`, `deadline_ts` — 모두 default 보존으로 backward compat
2. **`agora.instances` 응답 확장**: `inbox_depth`, `in_flight`, `last_seen_at`, `wait_mode` — 코드 추가 ~10줄
3. **`agora.wait` 반환에 `conversation_id`, `priority` 노출 + FIFO 보장 docstring**
4. **문서 사실관계 8건 패치** (위 D 표)
5. **운영 규약 명문화**: half-closed conversation의 120초 자동 cleanup, priority 인플레이션 방지("실제 차단 사유에만")

## H. 시그니처 통합안 (Inst4·Inst8·Inst7·Inst3 제안 합본)

```python
agora_dispatch(
    target: list[str],
    payload: dict,
    reply_to: str | None = None,
    in_reply_to: str | None = None,
    expect_result: bool = False,
    # --- 신규 ---
    conversation_id: str | None = None,        # None: in_reply_to 상속 또는 서버 UUID
    closing: bool = False,                     # 스레드 종결 의사
    priority: Literal['low','normal','high'] = 'normal',
    deadline_ts: str | None = None,            # ISO8601, advisory
) -> {id, conversation_id, created_at, target_inbox_depth_after}

agora_wait(
    timeout_ms: int | None = None,
    from_sources: list[str] | None = None,
    # --- 신규 ---
    by_conversation: str | None = None,
    sort: Literal['fifo','priority'] = 'fifo',  # priority: (priority desc, created_at asc, command_id asc)
) -> {commands: [{..., conversation_id, priority, wait_age_ms}]}

agora_instances() -> [{
    instance_id, role, description, registered_at,
    # --- 신규 ---
    inbox_depth: int,
    in_flight: int,
    last_seen_at: str,           # ISO8601
    wait_mode: 'auto' | 'manual' | 'unknown',
    accepting: bool,
}]
```

회귀 테스트 (Inst7 제안):
- 신규 필드 미지정 호출이 모든 응답에서 기존과 동일 (golden test)
- conversation_id 멀티홉 상속, 동시 교차의 별개 ID 생성
- closing 한쪽/양쪽 도달의 half/full closed
- priority 동률 + 같은 created_at에서 command_id 결정적 정렬
- 만료된 deadline_ts 처리 정책
